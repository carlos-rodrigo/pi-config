Pi Agent Memory Design

Agent Memory becomes a small Pi extension that recalls project and global lessons at prompt time, injects a bounded advisory memory pack for the current turn only, shows visible status, and learns cautiously from verification evidence.

  • Use before_agent_start as the recall boundary and append an ephemeral system-prompt pack; never persist recall packs as ordinary session messages.
  • Persist inspectable memory under .pi/agent-memory and $HOME/.pi/agent-memory; project results outrank global results and global promotion is human-gated.
  • Make trust visible through ctx.ui.setStatus(), /agent-memory commands, review logs, setup failure copy, and redaction/promotion gates.
  • Reuse semantic-search’s local Ollama patterns by extracting shared embedding/vector helpers only as needed; memory owns lessons and recall logs, not code indexes.
Start here

What reviewers should decide

Review the boundaries rather than implementation details: where memory enters the agent, where it is stored, what the user sees, and how a small first slice proves trust.

  1. Approve the prompt boundary. Memory packs are advisory system-prompt additions for one turn, not durable chat messages and not higher priority than system, user, project, or task instructions.
  2. Approve the storage and retrieval boundary. Two file-backed stores plus local embeddings satisfy the PRD without a database, daemon, or cloud vector service.
  3. Approve the first slice. Ship visible status, commands, and settings before recall so the default-on feature is inspectable from the beginning.
Non-blocking design knobs

Exact status copy and recall budget remain configurable constants until review chooses final wording and limits.

Example-first

Concrete system examples

The architecture optimizes for one trusted remembered turn and one broken setup that must be obvious.

Main scenario

The user runs /agent-memory add project "Use scripts/run_silent.sh for quiet verification.". Later, a matching verification task starts. The extension embeds the prompt locally, ranks the project memory above global memories, appends a small memory pack to the system prompt, logs recalled IDs, and shows mem: ready · project 1.

Setup failure scenario

Ollama or the required embedding model is unavailable. Memory does not silently skip and does not claim “no match.” It shows mem: failed · embeddings unavailable, writes a reviewable reason, injects no memory pack, and lets the user continue the task with clear recovery guidance.

Before

Useful conventions live in chat history, feature docs, semantic indexes, archives, and the user’s memory. Agents rediscover them or miss them.

After

Durable lessons are explicit, scoped, ranked, injected only when relevant, reviewed on demand, and visibly healthy or failed.

Chosen shape

Design thesis

Agent Memory should be an event-driven Pi extension with a small domain core, local file-backed stores, local embedding retrieval, advisory prompt injection, and reviewable feedback loops.

Use a Pi extension

Commands, lifecycle hooks, status, and verification event subscriptions are already extension concerns.

Inject ephemerally

Recall packs influence only the current turn and do not become durable chat/session state.

Persist inspectably

JSONL records and vector cache files under .pi satisfy local, reviewable, resettable memory.

System proposal

Architecture proposal

One extension coordinates delivery. Domain and storage code stay testable without Pi APIs, while embedding calls remain adapter-based.

Monorepo / package shape

Add extensions/agent-memory/ with tests and README. If implementation duplicates semantic-search embedding code, extract extensions/lib/semantic-retrieval.ts or equivalent.

Boundary: semantic-search owns code indexes; agent-memory owns durable lessons and recall logs.

Layers

Extension entrypoints call application services. Services call pure domain functions. Repositories and embedding adapters sit at the edge.

Dependency rule: domain code imports no Pi APIs and no Ollama client.

Runtime boundaries

All MVP work runs in the Pi process through hooks, commands, and event subscriptions.

Integration style: no daemon, background agent, local server, or cloud dependency.

Data ownership

Project and global stores are explicit scope roots. Settings, records, vector cache, and recall logs are written by the agent-memory extension only.

Persistence model: append-only records plus atomic settings/cache writes.

Implementation choices

Technology stack

Stay inside existing project technology so memory can be tested with fake ports and verified with the repo gate.

Pi APIs

before_agent_start, session_start, command registration, ctx.ui.setStatus(), custom/session messages for review output, and the Pi event bus.

Node filesystem

fs/promises, JSONL append files, atomic write-and-rename for settings and vector cache, an in-process per-scope write queue, and a stale-recoverable per-scope filesystem lock for concurrent Pi processes.

User-controlled Ollama

Use extensions/agent-memory/config.json defaults: a user-controlled remote Ollama server via SSH tunnel, mxbai-embed-large for atomic-memory recall, and qwen2.5-coder:14b for drafting/review helpers. No test should require a live model.

Interface implementation

Commands, status, and review output

The MVP interface is compact during normal work and detailed only when the user asks.

Composer/status component

Publish agent-memory status at startup, recall start, recall result, failure, pause, resume, disable, enable, and reset.

/agent-memory command family

One parser owns status|add|recall|review|pause|resume|disable|enable|reset so scope validation and help text stay consistent.

Review surface

Use command output or a custom review message to list recent recalls, candidates, metadata, and correction commands before building any overlay.

Accessibility and motion

Status copy must not rely on color alone. Review output should be text-first, command-addressable, and readable in TUI and non-TUI flows.

Architecture delta

What changes in the repo

The intended delta is narrow: one extension, optional shared retrieval helpers, status integration, and verification feedback consumption.

New extension

extensions/agent-memory/ owns entrypoints, services, domain rules, repositories, commands, status, and tests.

Optional shared helper

Status priority

Verification consumer

Boundary rules

Component communication boundaries

These rules keep the implementation small and testable.

Hooks and commands

Parse external Pi events and commands, then delegate to services. They should not rank memories or mutate record state directly.

Services

Coordinate status, stores, embedding adapter, recall service, prompt pack builder, and review output.

Domain

Owns precedence, eligibility, status transitions, safety gates, scoring rules, and promotion invariants.

Adapters

Filesystem and embedding adapters convert domain requests into local I/O. Tests replace them with fakes.

Domain model

Core memory concepts

The domain is about which lessons may influence a future prompt, not just storing text.

Memory record

An atomic lesson with scope, status, type, source reference, safety metadata, vector cache key, and usefulness evidence.

Scope store

Project and global stores own records and vectors. Project outranks global. Path scope is not modeled in MVP.

Recall event

Per-turn log of query summary, matched IDs, injected IDs, status, and setup/no-match/failure reason.

Review decision

User authority to activate, archive, edit, delete, reset, reject, restore, or promote memory.

Decisions

Design decisions and risks

The highest-risk choices are prompt influence, setup trust, and persistence. Each has an explicit boundary.

D1Ephemeral prompt patch

Chosen: append memory to the current system prompt in before_agent_start.

Rejected: hidden session messages that accumulate across turns.

D2Local embeddings required

Chosen: local adapter failures produce visible memory failure.

Rejected: cloud fallback or silent skip.

D3File-backed stores

Chosen: JSONL and vector cache files under scope roots.

Rejected: database or daemon for MVP volume.

D4Human-gated promotion

Chosen: only explicit user action creates active global memories from project lessons.

Rejected: automatic cross-project learning.

Prompt influence risk

Memory text must be presented as advisory. If memory conflicts with the user request, system instructions, AGENTS.md, PRD/design, or task brief, the higher-priority source wins and the conflict should be reviewable.

Slice designs

Outside-in task design and feedback hooks

Task briefs can be generated from these slice designs after design review. The first slice proves visible trust without retrieval complexity.

SLICE-001Status and controls shell

External need: the user sees memory running by default and can pause or disable it before any recall implementation ships.

Delivery surface

session_start publishes mem: running. /agent-memory status|pause|resume|disable|enable|reset return visible feedback.

Service/domain

The service turns command intent into domain state transitions: running to paused for the current session and enabled to disabled for persisted settings.

Persistence model

Disable/enable writes settings.json. Pause is session-local unless a later product decision changes that behavior.

Feedback hook: npm run test:agent-memory with fake Pi context verifies parser behavior, settings writes, and status labels.

Remaining slice design notes
  • SLICE-002: command-driven add writes a redacted scoped memory and asks the embedding adapter to cache a vector.
  • SLICE-003: prompt-time recall ranks project before global and appends an advisory memory pack.
  • SLICE-004: fake embedding failure proves visible failed state and no prompt injection.
  • SLICE-005: review commands write tombstones or approved promotion records.
  • SLICE-006: verification events update evidence only for same-task recalled IDs.
