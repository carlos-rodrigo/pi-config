Context Engineering

A durable design for how Pi configuration keeps agents aligned across sessions: short always-loaded instructions, on-demand docs, task-local feedback loops, and deterministic verification backpressure.

Key Takeaways

  • Persistent knowledge lives in AGENTS.md and docs/; operational task state lives in ignored .features/.
  • Agents load context progressively: baseline rules first, then feature docs, playbooks, task briefs, and subagents only when useful.
  • Verification is a backpressure layer: passing checks stay quiet; failures are injected back into the agent loop with actionable output.

02 · Direction

Design Thesis

The system is a layered context harness. Always-loaded instructions stay short; durable knowledge is stored in discoverable docs; task execution state is isolated in ignored task briefs; deterministic verification creates feedback pressure without filling the context window on success.

Chosen shape:

Use progressive disclosure plus deterministic checks, not a central knowledge graph, vector store, or chat-memory dependency.

03 · Baseline

Current Flow

The feature began from a failure mode: agents relied on chat memory, accumulated stale learning files, and produced noisy or skipped verification. The current repo has already implemented the replacement architecture, but these baseline risks explain why the design exists.

Context Loss

Without durable feature docs and concise entry instructions, an agent has to rediscover project rules or guess from stale conversation history.

Noisy Verification

Raw test output consumes context on success and can hide the one failure that matters when the system needs correction.

Unbounded Learning Files

Generic learning dumps do not reliably promote durable knowledge to the right place and eventually become low-signal context.

Mixed Durable And Operational State

Feature intent, architecture, task status, and verification evidence become hard to maintain when they live in the same artifact.

04 · Target

Intended Agent Flow

An agent should move from minimal persistent context to focused source-of-truth docs, then into a small implementation slice with explicit checks. It should only write durable docs when new reusable knowledge is discovered.

06 · Concepts

Domain Model

The domain model separates stable knowledge from work-in-progress state. This keeps review, compaction, and verification artifacts in the smallest useful scope.

Entry Instructions

Short, universal rules loaded at session start. In this repo, AGENTS.md names commands, search ladder, playbook loading rules, and verification behavior.

Playbook

On-demand reusable procedure under docs/playbooks/. It should be loaded only when the task matches the topic.

Feature Artifact

Durable source of truth for product or architecture context. Current design direction favors reviewable prd.html and design.html; this feature was sourced from legacy strategy.md/system-model.md.

Task Brief

Agent-readable, ignored Markdown under .features/ with goal, files, risks, feedback loop, and result evidence for one vertical slice.

Feedback Loop

A predeclared proof path: targeted checks, expected result, edge cases, manual check if needed, and final regression gate.

Context Firewall

Researcher/oracle agent jobs isolate broad search or second-opinion review so the parent session receives compact, decision-grade summaries.

07 · Decisions

Design Decisions

The primary tradeoff is intentional structure over automatic memory accumulation. The architecture chooses small explicit artifacts and deterministic hooks instead of broad implicit context.

D1 Progressive Disclosure Over Startup Dumps

Decision: Keep AGENTS.md compact and route recurring procedures to playbooks and skills.

Tradeoff: Agents must search/read the right docs, but startup context remains small and relevant.

D2 File-Based Docs Over Knowledge Graph

Decision: Use human-reviewable files in docs/ rather than building a custom vector store or feature-doc graph.

Tradeoff: Less automation, more direct reviewability and lower maintenance cost.

D3 Task-Local Results Over Durable Reports

Decision: Verification evidence belongs in task ## Result sections unless it changes durable knowledge.

Tradeoff: Completed task evidence is not part of the docs site, but durable docs stay concise.

D4 Silent Success, Verbose Failure

Decision: Verification output is swallowed on success and expanded on failure through scripts and lifecycle hooks.

Tradeoff: Humans get less celebratory output, agents preserve context for actual failures.

08 · Operations

Operational Concerns

This feature changes agent workflow rather than runtime user behavior. Operational quality is measured by whether agents find the right source, avoid noisy context, and cannot silently finish after breaking checks.

10 · Execution

Tasks And Feedback Hooks

The original implementation was split into nine task briefs under .features/context-engineering/tasks/. Future changes should create task briefs only when they need delegation, sequencing, or resumability.

Docs-Only Changes

Use targeted parse/link checks where possible, then run bash scripts/verify.sh --quick or full verification if touched project hooks should prove stability.

Extension Changes

Use verification_plan, run the relevant npm run test:<name>, then the repo gate. Verify command/tool/hook registration and malformed input behavior.

Skill/Workflow Changes

Check for stale terminology, read linked docs, and validate that task feedback loops still point to concrete commands and result evidence.

Final Gate

bash scripts/verify.sh is the release gate. It should remain silent on success and verbose on failure.
