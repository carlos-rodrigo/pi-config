Pi Agent Memory PRD

We want Pi to remember durable project and cross-project lessons by default, so agents can reuse verified context while the user can see whether memory is running, ready, paused, or failed.

  • Default experience is automatic and visible: memory runs by default, the composer/status component reports running/ready/failed states, and /agent-memory commands support pause/resume, disable/enable, review, reset, and manual creation.
  • Memory has two scopes: project memory in .pi/agent-memory outranks global memory under $HOME/.pi/agent-memory; project-to-global promotion requires explicit approval.
  • The product stores atomic lessons with source references, not raw transcripts; local embeddings are required, and broken setup fails visibly instead of being silently skipped.
  • Packet governance now requires every finished PRD to contain an authentic build-time Excalidraw semantic diagram; this legacy PRD remains migration-pending until its domain flow is rendered, inlined, and validated.
What we want to build

What

Pi Agent Memory is a local, inspectable memory layer for Pi sessions that recalls relevant project/global lessons automatically, can be seeded from day one, and learns cautiously from verified outcomes.

Automatic composer/status recall

By default, Pi retrieves a small bounded memory pack before a turn, uses it for the agent, and shows a compact composer/status indicator. Detailed recalled items are saved for review instead of being posted as a normal conversation message.

Atomic memory

Memory should progress from source references to atomic lessons, recurring scenarios, and profile-level preferences without losing traceability or storing raw transcripts.

Outcome feedback

Verification pass/fail and warning signals should strengthen or weaken recalled memories, so useful context gets more likely and misleading context fades.

Human review controls

The user can inspect, pause, disable, archive, edit, delete, reset, and promote memories. Project memories never become global automatically.

In scope
  • Default-on assisted automatic memory with visible pause/resume, disable/enable, and status controls.
  • Project and global memory scopes with clear labels and visible data locations.
  • User-created seed memories from the beginning via /agent-memory add.
  • Local-only recall using current semantic-search-style embeddings and lexical ranking.
  • Visible memory status, review, archive, reset, and promotion affordances.
Out of scope / non-goals
  • No custom path-scoped memory in the first release.
  • No external memory server, cloud vector database, or hidden third-party API dependency.
  • No silent global learning or autonomous self-modification.
  • No full raw transcript hoarding by default.
  • No short-term tool-log offload/Mermaid canvas in the first release.

Need and opportunity

Why

The user has many Pi commands, extensions, and project conventions. Manual memory commands would add friction, while no memory forces repeated explanations and missed lessons.

  • Need: Pi should remember verified workflows, decisions, gotchas, and preferences across sessions without daily user effort.
  • Current pain: Useful knowledge lives across chat history, feature docs, semantic-search indexes, archive records, and the user's head; agents must rediscover or ask again.
  • Opportunity: Use existing local semantic-search infrastructure plus Pi lifecycle events to make memory automatic, local, auditable, and feedback-weighted.
Success signals

Agents should naturally recall repo-specific test commands, known gotchas, and user preferences; the user should see less repeated explanation and still be able to inspect or correct memory.

Requirements and workflows

How

The product starts with default-on assisted automatic recall, explicit manual seeding for trust-building, and cautious learning from verified outcomes. Stories are intentionally framed around visible user/system behavior, not storage implementation.

STORY-001Run by default with visible controls

As the Pi user, I want memory to be running by default while clearly showing its state, so that I can benefit immediately and trust what happened.

  • REQ-001: The system must start with project and global memory enabled by default. The first visible notice must explain active scopes, data locations, the local embedding requirement, and controls such as /agent-memory pause, /agent-memory resume, /agent-memory disable, /agent-memory enable, /agent-memory status, /agent-memory review, and /agent-memory reset.
  • REQ-002: The system must show a concise composer/status indicator indicating whether memory is running, recalling, ready, failed, paused, or disabled.

STORY-002Recall relevant context automatically

As an agent working in Pi, I want relevant project and global memory before I act, so that I can follow proven local conventions and cross-project lessons.

  • REQ-003: The system must recall from project memory before global memory and label each recalled item by scope. Custom path-scoped memory is not part of the MVP.
  • REQ-004: The system must use a bounded memory budget. No relevant memories should produce a visible empty/none state; unavailable local embeddings or recall setup failures must produce a visible memory failure rather than a silent skip.

STORY-003Learn from verified outcomes

As the Pi user, I want memory usefulness to be reinforced by real task outcomes, so that repeated successes become more available and bad memories lose influence.

  • REQ-005: Passing verification may boost recalled memories; failing verification may weaken them and propose a candidate gotcha.
  • REQ-006: Project-safe atomic lessons may be created from verified outcomes from the beginning; global memories, preferences, sensitive/client-specific details, raw excerpts, and failure-only lessons must require review before affecting future work.

STORY-004Inspect and correct memory

As the Pi user, I want lightweight review controls when memory is wrong or noisy, so that I can archive, edit, delete, reset, or promote memories without digging through hidden state.

  • REQ-007: The system must provide a visible review surface for recent recalls, candidates, global promotion requests, metadata, and correction actions.
  • REQ-008: Archived, deleted, rejected, or reset memories must not be recalled or scored again unless the user restores or recreates them.

STORY-005Create memories from the beginning

As the Pi user, I want to seed memory explicitly while I test the feature, so that I can build trust by seeing a memory I created get recalled later.

  • REQ-009: The user must be able to create an atomic project or global memory manually with a command such as /agent-memory add project "..." or /agent-memory add global "...".
  • REQ-010: Memory must store atomic lessons and source references, not full prompt/session transcripts or raw tool output.
  • REQ-011: Memory creation must redact secret-like values such as API keys, tokens, passwords, and Authorization headers; sensitive/client-specific content becomes review-only unless the user explicitly approves the final text and scope.

WF-001Main workflow

Given memory is running by default and a project has stored lessons. When the user sends a normal task prompt. Then Pi shows a compact composer/status recall state, injects the bounded scoped memory pack for the agent, and verification later updates usefulness.

WF-002Bootstrap workflow

Given a new project or install has no useful memories yet. When the user runs /agent-memory add project "..." or approves a project-safe candidate. Then Pi stores an atomic memory, confirms the scope, and can recall it on a later matching turn.

WF-003Setup failure workflow

Given local embeddings are unavailable or recall setup is broken. When a turn starts. Then Pi shows a visible memory failed state with a reviewable reason and recovery path instead of silently skipping memory.

After every story

Post-Story User Flows

Each story keeps the normal Pi workflow intact: the user types a task, sees compact memory behavior in the composer/status area, can seed memory deliberately, and can inspect or correct memory later.

FLOW-001Default-on memory, then work normally

  1. Entry

    The user starts in Pi after installing or updating the memory extension.

  2. Action

    Pi starts memory automatically for project and global scopes and shows the composer/status component.

  3. Visible response

    Pi shows memory mode, active scopes, data locations, local-only policy, setup health, and how to pause/resume, disable/enable, review, or reset memory.

  4. Next decision

    The user returns to normal prompts, pauses with /agent-memory pause, or opens details/review if the status looks wrong.

  5. Resolved decision

    Memory is enabled by default and uses two scopes: project and global.

FLOW-002Automatic status recall

  1. Entry

    The user asks Pi to change or investigate a feature.

  2. Action

    Pi searches project and global memory using the prompt and current working directory.

  3. Visible response

    A composer/status indicator shows “memory running”, “recalling”, “memory ready”, “no memory matched”, or “memory failed”. Detailed scoped bullets are saved in the memory/review log instead of appearing as a normal conversation message.

  4. Next decision

    The user can ignore a ready/empty state, open details, pause memory, or fix setup when the component reports failure.

  5. Remaining choice

    Choose the default memory budget and exact composer/status copy.

FLOW-003Verified learning

  1. Entry

    A task finishes and verification emits pass/fail evidence.

  2. Action

    Pi updates the usefulness of recalled memories and may create or draft a project-safe atomic lesson from verified failure/success patterns.

  3. Visible response

    Scoring updates stay quiet; newly saved project lessons and review-only candidates appear in the memory/review log instead of silently becoming global.

  4. Next decision

    The user can approve, edit, reject, delete, ignore, or promote a candidate later.

  5. Resolved decision

    Project-safe atomic lessons may be created from the beginning; global, sensitive, preference, raw-excerpt, and failure-only lessons wait for review.

FLOW-004Memory review

  1. Entry

    The user notices a bad recall, sees memory noise, or wants to promote a useful project lesson.

  2. Action

    The user opens memory review with /agent-memory review or the composer/status details affordance.

  3. Visible response

    Pi lists recent recalls, candidates, scopes, source evidence references, timestamps, confidence/usefulness evidence, and actions.

  4. Next decision

    The user archives, edits, deletes, resets, promotes, rejects, or leaves memory unchanged.

  5. Resolved decision

    MVP review uses /agent-memory review plus composer/status details; a fuller TUI overlay can come later.

FLOW-005Manual seed memory

  1. Entry

    The user wants to test memory with a known lesson before enough verification history exists.

  2. Action

    The user runs /agent-memory add project "Use scripts/run_silent.sh for verification output" or chooses global for a cross-project lesson.

  3. Visible response

    Pi confirms the atomic lesson, scope, source reference, redaction status, and whether it is active or waiting for review.

  4. Next decision

    The user can run a matching prompt to see it recalled, edit it, or delete/reset it if the recall is noisy.

Domain behavior

How Memory Concepts Interact

The product domain is memory formation and recall, not code search itself. Memory should be useful because it is scoped, ranked, traceable, and visibly healthy.

DOMAIN-INT-001Prompt recalls scoped memories

User prompt + cwd → recalls → project/global memory pack

Rule / authority: Project memory from .pi/agent-memory outranks global memory from $HOME/.pi/agent-memory. No custom path scope is included in the MVP.

Vocabulary gap: Confirm whether the top layer is called “profile”, “persona”, or “agent profile” in Pi.

DOMAIN-INT-002Outcome updates usefulness

Verification outcome → scores → recalled memory salience and confidence

Rule / authority: Scoring can update automatically only for memories that were recalled in the same task context.

Vocabulary gap: Choose whether user-facing copy says “boost/weaken”, “trusted/less trusted”, or “hit/miss”.

DOMAIN-INT-003Project memory promotes to global only by approval

Project lesson → candidate → user approval → generalized global memory

Rule / authority: No project-specific fact, file path, client detail, or sensitive data becomes global without a visible approval step.

Vocabulary gap: Decide whether “promote global” is a standalone action or part of candidate review.

DOMAIN-INT-004User creates atomic memory

User-authored lesson → redacts + scopes → active project/global memory or review candidate

Rule / authority: The user can seed memory manually from day one. Project memories can become active for the current project; global memories require either explicit global creation or approved promotion.

Visible metadata: scope, type, atomic lesson, source reference, status, usefulness/confidence evidence, created/updated timestamps, and last recalled time.

Approved artifact invariant

PRD Diagram Governance

The packet may not claim finished validation without at least one meaningful semantic diagram rendered by the build-time Excalidraw pipeline.

Hard invariant

Every finished prd.html must include at least one authentic Excalidraw diagram in user-flows or domain-interactions. A UI mockup decision may avoid redundant option mockups, but it cannot waive the semantic diagram. HTML/CSS wireframes do not satisfy this invariant.

Validator scope

The hard gate applies to finished files named prd.html. It must require an inline <svg role="img">, the svg-source:excalidraw marker, stable data-review-id anchors, accessibility text, and preserved diagram-reveal groups. PRD templates validated with --allow-placeholders need a canonical diagram slot but not a rendered feature diagram; design and other report checks remain unchanged.

Migration criteria

A legacy PRD is compliant only when its source scene is retained near the feature packet, the rendered SVG is inlined into the relevant semantic section, existing useful HTML/CSS wireframes remain intact, and the finished report passes validate-html-report.mjs without placeholder mode. There is no silent grandfathering: an unmigrated PRD may remain readable, but it cannot claim validated or ready status.

Current packet status

docs/features/agent-memory/prd.html is the first migration fixture. Its text domain cards and UI wireframes remain useful, but they do not yet satisfy the approved invariant. Enforcement and migration are assigned to the next implementation task; no enforcement code changed in this packet update.

Reviewer choices

Review Gaps

The blocking product choices from review have been resolved. Remaining gaps are naming/copy polish that should not block architecture design.

DEC-001Default-on scopes and storage visibility

Decision: Memory runs by default with two scopes: project memory in .pi/agent-memory for the current workspace and global memory under $HOME/.pi/agent-memory for this Pi install. Project memory outranks global memory.

Impact: Startup/status copy must show active scopes, data locations, and pause/resume, disable/enable, review, and reset controls.

Owner/status: User reviewed; resolved for design.

DEC-002Memory creation from the beginning

Decision: The user can create seed memories immediately with /agent-memory add project "..." or /agent-memory add global "...". Project-safe atomic lessons may also be saved from verified outcomes; global promotion, personal/user preferences, sensitive or client-specific details, raw transcript excerpts, large code snippets, and failure-only lessons require review before recall.

Impact: Design must include manual add, redaction, review-only candidate states, and visible confirmation of active scope.

Owner/status: User reviewed; resolved for design.

DEC-003Visibility and failure posture

Decision: Recall should not appear as a normal conversation message. Show a composer/status component with running, recalling, ready, no-match, failed, paused, and disabled states. Missing local embeddings or broken setup must fail visibly instead of silently skipping memory.

Impact: Keeps the conversation clean while preserving auditability and setup trust.

Owner/status: User reviewed; resolved for design.

DEC-004MVP review surface

Decision: Use /agent-memory review plus the composer/status “open details” affordance for recent recalls, candidates, and correction actions. Defer a full TUI overlay until after MVP.

Impact: Design should prioritize the command/detail surface and not make a complex overlay a release blocker.

Owner/status: User reviewed; resolved for design.

GAP-005L2/L3 terminology

Question: Should the top memory layer be called persona, profile, agent profile, or something else in Pi?

Impact: Changes labels and mental model but not core behavior.

Owner/status: No change requested; keep for design naming polish only.

Acceptance Criteria

Acceptance is based on what the user and agent can observe in normal Pi use.

AC-001Default-on visible state

Given Pi starts after install or update, when the memory extension loads, Pi shows memory running by default, active project/global scopes, data locations, local-only behavior, and how to pause/resume, disable/enable, review, or reset it.

AC-002Composer/status automatic recall

Given relevant memory exists, when a normal task prompt starts, Pi shows a composer/status recall state and saves detailed scoped memories for review rather than posting them as a normal conversation message.

AC-003Scope precedence

Given both project and global memories match, project memories rank first in the injected memory pack and review details; global memories are labeled separately.

AC-004Outcome-weighted memory

Given a memory was recalled during a task, when verification passes or fails, Pi updates the memory's usefulness evidence without creating hidden global memories.

AC-005Human-gated global promotion

Given a project lesson looks reusable, when Pi proposes promotion, the user can approve, edit, or reject before it affects other projects.

AC-006Inspectability

Given the user asks what memory influenced a turn, Pi can show recent recalled items, scope, source evidence reference, metadata, and available correction actions.

AC-007Manual seed memory

Given the user runs /agent-memory add project "..." or /agent-memory add global "...", Pi confirms the atomic lesson, scope, source/redaction status, and whether the memory is active or waiting for review.

AC-008Setup failure is visible

Given memory is running but local embeddings are unavailable or recall setup is broken, when a turn starts, Pi shows a memory failed state with a reviewable reason and recovery path instead of silently skipping memory.

AC-009Pause and disable controls

Given the user runs /agent-memory pause or /agent-memory disable, when the next turn starts, no memory pack is injected and Pi shows paused/off with a resume or enable path.

AC-010Review correction and reset

Given the review surface, the user can archive, edit, delete, reset, promote, or reject a memory; subsequent recall honors that decision.

AC-011Hard PRD diagram gate

Given a finished PRD is validated, when it lacks an authentic build-time Excalidraw diagram in its user-flow or domain-interaction content, validation fails with an actionable migration message; after the semantic scene is rendered and inlined with accessibility, review anchors, and reveal metadata, validation passes without replacing useful HTML/CSS wireframes.
