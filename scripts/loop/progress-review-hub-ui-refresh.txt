## Codebase Patterns

- Frontend workspace lives in `extensions/review-hub/web-app`; invoke scripts from extension root via `npm --prefix web-app ...`.
- Keep shadcn component aliases aligned (`@/*` → `src/*`) across `tsconfig.app.json`, `vite.config.ts`, and `components.json`.
- `lib/server.ts` now prefers `web-app/dist` and falls back to legacy `web/`; treat `/manifest.json`, `/audio`, `/visual*`, `/comments*`, and `/complete` as reserved API routes before SPA fallback.
- Frontend API types are imported from backend contracts via `web-app/src/lib/api/types.ts` (type-only import from `../../../../lib/manifest`).
- Layout is split into reusable rails (`components/layout/toc-rail.tsx`, `components/layout/comment-rail.tsx`) and controlled from `App.tsx` mode state (`read` | `review`).
- `VisualContentHost` is the integration point for `/visual` + `/visual-styles`, with `IntersectionObserver` driving active TOC sync and imperative `scrollToSection` for TOC clicks.
- Comment rail is composed from `components/comments/*` (filters, composer, list) so behavior updates can be isolated without rewriting shell layout.
- `/comments` is an upsert endpoint: server owns `createdAt`/`updatedAt` and normalizes missing status to `open`; status toggles should send full comment payload + desired `status`.
- Keep unresolved workflow logic in pure utilities (`web-app/src/lib/unresolved-navigation.ts`, `web-app/src/lib/unresolved-shortcut.ts`) and let hooks/components compose them for easier testing.
- Audio UX should be split into pure helpers (`web-app/src/lib/audio-player.ts`) plus a runtime hook (`hooks/useAudioSync.ts`) so state messaging and timestamp mapping stay testable.
- For server-rendered visual controls, prefer inline SVG markup over emoji glyphs so iconography stays consistent and grep-able for migration checks.
